## user

请分析以下项目，并生成一份详细的项目架构文档：

**当前文件**: `${context.filename}`

**文件类型**: `${context.filetype}`

### 分析步骤：

1. 使用 @{file_search} 扫描项目根目录，识别项目类型（通过 package.json、Cargo.toml、go.mod、CMakeLists.txt、pom.xml 等标志文件）
2. 使用 @{file_search} 分析目录结构，排除依赖目录（如 node_modules/、vendor/、target/、build/、dist/、.git/），列出核心源码目录
3. 读取关键的构建/配置文件（如步骤1识别的文件、Dockerfile、docker-compose.yml 等），理解项目依赖和技术栈
4. 基于目录结构，识别核心模块/服务边界，使用 @{read_file} 读取各模块的入口文件或 README 以确认职责
5. 若项目规模适中，使用 @{grep_search} 分析模块间的导入/引用关系，梳理依赖图
6. 将最终分析结果通过 @{memory} 保存，键名为 `project_map_${project_name}`

### 分析要求：

1. **项目概览**
   - 项目类型、语言及主要框架/技术栈
   - 整体目录结构（树形展示，深度建议 2-3 层）
   - 关键配置文件及其作用

2. **模块/组件分析**
   - 核心模块/服务的职责说明
   - 模块间的依赖关系（可用文本层级或 Mermaid 图表示）
   - 各模块关键文件清单

3. **数据流/控制流**
   - 主要数据流向（如请求处理流程、事件驱动链路）
   - 关键业务流程的入口和核心步骤

4. **输出格式**
   - 使用 Markdown 格式，层级清晰
   - 代码块标注语言类型
   - 复杂关系建议使用 Mermaid 图表

### 约束条件：

- 若文件数量超过 20 个，优先分析核心模块，避免全量遍历
- 遇到不确定的目录用途时，优先读取该目录下的 README 或主文件确认
- 精确优先，避免冗余：用最少必要信息传达核心架构
- 在确保描述精确的前提下，力求简洁，避免冗余信息
