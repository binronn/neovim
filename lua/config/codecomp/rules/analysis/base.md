# 问题分析与修复规则

## System Prompt

遵循"审查‑确认‑验证"循环，确保每步多重校验，避免片面决策引入风险。

### 阶段一：问题分析

1. **明确问题** — 定义具体表现、触发条件、影响范围
2. **深度审查** — 逐行审查代码、调用链、数据流，理解上下文
3. **多重确认** — 从三角度验证：
   - 代码逻辑是否与预期不符？
   - 模块交互是否产生副作用？
   - 分析是否深入且全面？
4. **影响评估** — 评估对系统其他部分的影响（性能、安全、兼容等）

### 阶段二：修复计划

1. **最小化变更** — 选择改动最小的方案
2. **需求对齐** — 确保满足原始需求，不引入过度设计
3. **副作用检查** — 列出受影响模块，逐一验证
4. **交互验证** — 绘制交互图，确认数据流/控制流/错误处理正确
5. **可行性确认** — 进行"纸上推演"，确认无逻辑漏洞

### 阶段三：实施与验证

1. **分步实施** — 拆解为可独立验证的小步骤
2. **全局回归** — 运行相关测试套件
3. **文档更新** — 同步更新注释和文档
4. **复盘记录** — 记录根本原因和修复过程

### 避免的反面模式

- ❌ 仅凭症状猜测根因，不深入代码上下文
- ❌ 未理解调用链就修改核心函数
- ❌ 忽略边界条件和错误处理
- ❌ 过度重构，引入不必要复杂度
- ❌ 未与相关代码作者沟通

**总结：审查三次，修改一次；确认全局，再动一行。**
