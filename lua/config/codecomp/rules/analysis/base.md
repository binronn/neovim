# 问题分析与修复规则

## 核心原则

在处理任何代码问题或需求时，必须遵循“审查‑确认‑验证”的循环流程，确保每一步都经过多重校验，避免因片面或仓促的决策引入风险。

## 阶段一：问题分析

1. **明确问题** — 在开始分析前，必须用清晰的语言定义问题的具体表现、触发条件和影响范围。
2. **深度审查** — 对问题相关的每一处代码、每一个调用链、每一个数据流进行逐行审查，理解其上下文和依赖。
3. **多重确认** — 至少从三个角度验证问题确实存在：
   - 代码逻辑是否与预期行为不符？
   - 相关模块的交互是否产生了副作用？
   - 分析是否足够深入，且分析面是否足够全面
4. **影响评估** — 明确问题对系统其他部分的影响，包括性能、安全性、兼容性等。

## 阶段二：修复计划

1. **最小化变更** — 优先选择能够解决问题且改动范围最小的方案。
2. **需求对齐** — 确保修复方案完全满足原始需求，不引入过度设计或无关功能。
3. **副作用检查** — 列出所有可能受影响的模块和函数，逐一验证修复不会破坏现有功能。
4. **交互验证** — 若修复涉及多个模块的交互，必须绘制交互图，确保数据流、控制流和错误处理均保持正确。
5. **可行性确认** — 在最终确定方案前，至少进行一次“纸上推演”，模拟修复后的执行流程，确认无逻辑漏洞。

## 阶段三：实施与验证

1. **分步实施** — 将修复拆解为可独立验证的小步骤，每完成一步立即进行单元测试。
2. **全局回归** — 修复完成后，运行所有相关测试套件，确保没有引入回归错误。
3. **文档更新** — 同步更新代码注释、接口文档或用户手册，保持文档与代码一致。
4. **复盘记录** — 将问题的根本原因、修复方案和验证过程记录在项目知识库中，供后续参考。

## 必须避免的反面模式

- ❌ 仅凭症状猜测根本原因，未深入代码上下文。
- ❌ 在未完全理解调用链和依赖关系的情况下修改核心函数。
- ❌ 忽略边界条件和错误处理，导致修复在极端场景下失效。
- ❌ 为了“完美”而过度重构，引入不必要的复杂度和风险。
- ❌ 未与相关模块的负责人或代码作者沟通，导致修复方案与整体架构冲突。

## 一句话总结

**“审查三次，修改一次；确认全局，再动一行。”**
